# Code Context: ydd (Яндекс.Директ Дэшборд)

## Files Retrieved

1. `dashboard.html` (1–594) — единственный файл приложения (SPA). Всё: разметка, стили, скрипты.
2. `2026-05-19_2026-05-19_immortal223.xlsx` (~347 KB) — файл-образец данных.
3. `.openclaude/settings.local.json` — разрешения: curl, python, WebFetch(yandex.ru).

## Архитектура

**Pure client-side SPA. Никакого бэкенда, сборщика, зависимостей, кроме CDN.**

### Поток данных

```
File (drag/click)
  → FileReader.readAsArrayBuffer(file)
    → XLSX.read(buffer, {type:'array'})        // SheetJS CDN
      → XLSX.utils.sheet_to_json(ws, {header:1}) // [[cell,...],...]
        → parseData(rows)
          1. Ищет headerRow в первых 15 строках (по сигнатурам «кампания», «показы», «клики»)
          2. Парсит rows[headerIdx+1 … end], пропуская строки-резюме («с …», «Итого», «Всего»)
          3. Вызывает detectColumns() → auto-определение dim/metric
          4. Извлекает период из первых строк (период / дата)
          5. buildUI() → buildFilters() + buildContent()
          6. applyFilters() → renderAll()
```

### Ключевые функции и связи

```
handleFile(e)
  └→ parseData(rows)
       ├→ detectColumns()          — опреление типов колонок через COL_META + fallback
       ├→ buildUI()
       │    ├→ buildFilters()      — <select> для каждой dimension (кроме id; ≤500 значений; топ-30 для регионов)
       │    └→ buildContent()      — KPI grid + charts grid + table (пустые контейнеры)
       └→ applyFilters()
            └→ renderAll()
                 ├→ renderKPIs()    — 12 карточек (приоритет: показы/клики/расход/CTR/конверсии/отказы)
                 ├→ renderCharts()
                 │    ├→ destroy всех старых chart-экземпляров
                 │    ├→ aggregateBy(primaryDim, filteredData)
                 │    └→ создание до 8 chart'ов:
                 │         1. Bar: spend + clicks (двойная ось)
                 │         2. Bar: CTR %
                 │         3. Bar: CPC
                 │         4. Doughnut: spend по dim (2-10 значений)
                 │         5. Bar: bounce rate (цветовая маркировка)
                 │         6. Bar: конверсии + цена цели
                 │         7. Bar (horizontal): топ-15 регионов
                 │         8. Bubble: CTR vs CPC
                 └→ renderTable()
                      └→ aggregateBy(primaryDim) → сортировка → HTML

aggregateBy(dimCol, data)
  └→ Итерирует все строки, группирует по dimCol, суммирует/усредняет метрики
     Учитывает calc: ctr (Показы/Клики*100) и wctr (взвешенный CTR)
```

### COL_META — мета-словарь колонок

Около 70 известных колонок Яндекса с типами:
- `dim` — измерения (кампания, группа, устройство, регион, пол, возраст и т.д.)
- `metric` — метрики с `agg: sum|avg|calc` и `fmt: int|money|pct|num2`

Fallback для неизвестных колонок — эвристика по первым 100 строкам:
- если <30% чисел → dimension
- если все значения 0-100 → `pct`
- иначе → `int`

### Обработка ошибок: что есть

| Что есть | Где |
|---|---|
| `alert(...)` если заголовки не найдены | `parseData`, строка ~159 |
| `try`/`catch` **отсутствуют** | весь код |
| Пропуск пустых/служебных строк | `parseData` (проверка `first.startsWith('с ')` и т.д.) |
| `hasValue(v)` проверка `undefined/null/''/NaN` | используется в `aggregateBy` и `renderKPIs` |
| `toNum(v)` с защитой от NaN | хелпер |
| `formatValue(val, fmt)` с fallback `'—'` | хелпер |
| `hidden` класс для empty state | показать/скрыть при загрузке |

### Чего не хватает в обработке ошибок

- Нет `try/catch` вокруг `XLSX.read()` — битый файл убьёт страницу
- Нет `try/catch` вокруг `new Chart()` — при проблемах с canvas график просто сломается
- Нет индикатора загрузки (spinner) для больших файлов
- Нет обработки случая, когда `metrics` пуст
- Нет проверки `rawData.length === 0` после парсинга
- `alert()` — единственный способ сообщить об ошибке, нет UI-нотификаций

### Точки интеграции

| Интеграция | Место | Риск |
|---|---|---|
| **FileReader** (Web API) | `handleFile`, строка ~133 | Утечек нет (одноразовый), но код синхронный после onload |
| **SheetJS** (XLSX CDN) | `XLSX.read()`, `XLSX.utils.sheet_to_json()` | Только первый лист `wb.SheetNames[0]`. CDN-зависимость |

## Слабые места, ограничения, баги

### Производительность (большие данные)

1. **Всё в памяти + синхронная отрисовка.** `aggregateBy` итерирует ВСЕ строки на каждый фильтр. Для 100k+ строк — freeze UI.
2. **Нет виртуализации таблицы.** `renderTable` строит всю строку HTML для всех групп. При 500+ строк таблицы — медленно.
3. **renderCharts** вызывается при каждом `applyFilters` и пересоздаёт до 8 графиков с нуля.
4. **filterSelect** для dimension с ≤500 значений — `<option>` для каждого значения. При 500 опциях — громоздкий DOM.
5. **`getUniqueValues`** итерирует все строки для каждой колонки при `detectColumns` — O(N*M).

### Утечки

1. **`chart.destroy()` — только в `renderCharts()`** при ререндере. Если пользователь уйдёт со страницы до нового рендера, chart-объекты остаются в памяти.
2. **Нет `removeEventListener`** на `document.getElementById('fileInput').addEventListener('change', handleFile)` — хранится навсегда.
3. Глобальные переменные (`rawData`, `filteredData`, `charts`) — при повторной загрузке файла старые данные и графики корректно заменяются.

### Баги / неточности

1. **Парсинг периода** — ищет строку с `'период'` или `\d{2}.\d{2}.\d{4}` среди первых `headerIdx` строк. Если период в подвале — не найдёт.
2. **`sortCol` инициализируется** при первом вызове `renderTable()` метрикой с `fmt: 'money'`, но если её нет — падает с `cannot read property 'name' of undefined`.
3. **Топ-30 регионов** — фильтр строится по топ-30 по расходу, но если выбран другой фильтр — список не обновляется.
4. **Кнопка «Сбросить фильтры»** не сбрасывает `sortCol`/`sortDir` — таблица остаётся отсортированной.
5. **Одинаковые названия метрик** (например, «Расход (руб.)» и «Расход, руб.») — KPI создаёт две карточки с обоими.
6. **Doughnut палитра** — `PALETTE` (20 цветов) циклически повторяется. При >20 сегментов цвета начнут повторяться.
7. **Агрегация CTR** — считает `imp/clk*100` только для метрики с `calc: 'ctr'`. Если колонка называется «CTR (%)» и не имеет `calc` — будет среднее, а не суммарное отношение.

### Не реализованные метрики/фичи

- **Сравнение периодов** (MoM/YoY) — полностью отсутствует
- **Date range picker** / фильтр по дате
- **Экспорт** (изображение, PDF, CSV)
- **Сохранение состояния** (localStorage)
- **Multi-select** фильтры
- **Поиск/автокомплит** в фильтрах (особенно для регионов с 500+ значений)
- **Сводка по кампании** (drill-down при клике)
- **Sparklines/мини-графики** в KPI
- **Нормализация/стандартизация метрик** (расход/конверсия на 1000 показов)
- **Тёмная/светлая тема** (есть только тёмная)
- **Интерактивность** (tooltip у графиков — только стандартный Chart.js)

### Мобильная версия / Доступность

- **`@media (max-width: 900px)`** — только переключение charts-grid на 1 колонку и KPI на `minmax(140px, 1fr)`.
- Нет **touch-оптимизации** для фильтров (<select> маленькие).
- Таблица **не прокручивается по горизонтали на мобильных** — `overflow-x: auto` есть, но ширина колонок не адаптируется.
- Нет **aria-атрибутов**, **role**, **keyboard navigation**.
- **Контрастность**: тёмная тема с `--text: #e4e6f0` на `--surface: #1a1d27` — проходной контраст, но мелкий текст `--text2: #8b8fa8` на тёмном фоне может быть трудночитаем.

## Готовность к расширению

### Добавление нового типа графика

**Легко** (10-15 минут):

1. Найти подходящую метрику (или пару)
2. Вставить блок в `renderCharts()`:
   ```js
   if (someMetric) {
     const card = makeChartCard('Заголовок', 'badge');
     grid.appendChild(card);
     charts.myChart = new Chart(card.querySelector('canvas'), {
       type: '...',
       data: { ... },
       options: { ... }
     });
   }
   ```
3. При необходимости — новый `chartOpts*` helper.

**Ограничения:**
- `PALETTE` — 20 цветов, при >20 сегментов цикл
- `aggregateBy` возвращает плоский объект с label + метрики; для сложных агрегаций (например, stacked с несколькими dim) нужно писать отдельную логику
- Нет единого store/state — все данные глобальные

### Добавление бэкенда / API Яндекс.Директа

**Средняя сложность.**

Что потребуется:
1. **Прокси/CORS** — Яндекс.Директ API не разрешает CORS-запросы с браузера. Нужен бэкенд-прокси.
2. **OAuth 2.0** — Яндекс.Директ требует токен.
3. **Маршрутизация** — текущая архитектура не имеет роутинга (нет флагов загрузки из API vs файла).
4. **Кеширование** — текущий код не умеет кешировать ответы.

Архитектурное решение: минимальный прокси-сервер (напр. Express/Fastify), который делегирует в Report API Яндекс.Директа. Dashboard.html остаётся SPA, но вместо `handleFile` добавляется `fetchCampaigns()`.

### Добавление сравнения периодов

**Средняя сложность.**

Проблемы:
1. `aggregateBy` не умеет разбивать данные по времени — нужно добавить группировку `dim × period`.
2. Нет date-парсинга: колонка «Дата» определена как dim, но не приведена к Date.
3. UI для выбора двух периодов — нужно добавить date inputs.
4. Рендеринг: `renderKPIs` и `renderCharts` ожидают один набор данных. Для сравнения нужны два прохода `aggregateBy` с разными фильтрами.
5. Визуализация: grouped bar вместо single bar, delta-колонки в таблице.

**Рекомендация:** начать с парсинга «Дата» в `parseData` (преобразовать в Date), затем добавить `periodFilter` в `applyFilters`, потом — два date-range пикера и рендеринг с diff.

## Открытые вопросы (требуют уточнения)

1. **Откуда будут браться данные в будущем?** Только XLSX-файлы (текущий вариант) или нужна интеграция с API Яндекс.Директа? Если API — какой именно отчёт (Report API v1?).
2. **Какой типичный объём данных?** 10³, 10⁴, 10⁵ строк? От этого зависит необходимость виртуализации и оптимизаций.
3. **Нужна ли поддержка нескольких листов/файлов?** Сейчас читается только первый лист первого файла. Нужен ли multi-upload?
4. **Какие метрики реально нужны?** Текущие 8 графиков — избыточны или неполны? Какие топ-3 графика используются чаще всего?
5. **Требуется ли сравнение периодов (WoW/MoM/YoY)?** Если да — как их выбирать (календарь, предустановленные интервалы)?
6. **Нужна ли светлая тема?** Или тёмная — окончательный вариант?
7. **Планируется ли бэкенд?** Или приложение остаётся чисто клиентским? Если бэкенд — какой стек предпочтителен?
8. **Кто целевая аудитория?** Только пользователь или публичный дашборд? Нужна ли мультиязычность?
9. **Как обрабатывать валюты/регионы?** Яндекс.Директ может выдавать данные в разных валютах. Нужна ли конвертация?
10. **Нужен ли экспорт** (PDF, PNG, CSV) или визуализация в браузере — конечная цель?

## Start Here

**Файл:** `dashboard.html` (весь файл, 594 строки)

Это единственный файл приложения. Любое изменение начинается с понимания его структуры. Приоритетная зона для первого изменения:

- **Если срочный баг**: функция `renderCharts()` (строки 320–480) — именно там создаются все графики и destroy старых.
- **Если новая фича**: `parseData()` (строки 138–190) — расширение форматов входных данных; или `renderCharts()` для нового графика.
- **Если производительность**: `aggregateBy()` (строки 260–295) — узкое место при больших данных.
