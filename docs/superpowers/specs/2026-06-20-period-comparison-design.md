# Спецификация: Сравнение периодов (два файла А vs Б)

**Дата:** 2026-06-20
**Файл:** `dashboard.html` (единственный файл приложения)
**Режим:** расширение существующей single-file SPA, без бэкенда

## Контекст и цель

Дашборд Яндекс.Директа (`dashboard.html`, ~920 строк) — чисто клиентская SPA на Chart.js + SheetJS. Сейчас отображает один загруженный XLSX-файл: KPI, 8 графиков, детальная таблица. Главная отсутствующая аналитическая возможность — **сравнение периодов**: понять, как изменились метрики между двумя выгрузками (например, прошлый месяц vs текущий).

Цель итерации — добавить режим сравнения двух файлов (А — основной, Б — для сравнения) с расчётом дельты по всем метрикам, плюс точечные фиксы надёжности, критичные для корректности сравнения.

## Подход (выбран из трёх)

**Подход A — Параллельные датасеты.** Каждый файл парсится в независимый набор (`datasetA`, `datasetB`). Существующий single-режим не меняется. При включённом тумблере и двух загруженных файлах включается режим `compare`, в котором агрегаты двух датасетов сопоставляются и считается дельта.

Отвергнутые альтернативы:
- **B (слияние с тегом периода)** — загрязняет общее состояние, ломает single-режим, дельту всё равно считать post-process'ом.
- **C (два независимых рендера)** — не считает дельту, ради которой всё затевается.

## Скоуп

### Входит

1. Режим сравнения двух файлов с дельтой по метрикам.
2. Тумблер «Сравнение периодов» в header.
3. Рендер сравнения: KPI с дельтой, 3 графика, таблица с Δ.
4. Баг-фикс: CTR-агрегация по role, а не по жёсткому имени колонки.
5. Баг-фикс: toast-нотификации вместо `alert()`.

### Не входит (явно, отложено в следующие фазы)

- Палитра doughnut при >20 сегментах.
- Spinner / индикатор загрузки для больших файлов.
- Защита `sortCol` при отсутствии money-метрики.
- Виртуализация таблицы, кеширование `aggregateBy`, дебаунс фильтров.
- Светлая тема, ARIA, keyboard navigation.
- Сохранение состояния тумблера в localStorage.
- Экспорт (CSV/PNG/PDF), drill-down, multi-select фильтры, временные ряды.

## Архитектура и состояние

### Новое состояние (расширение блока `STATE`)

```js
let mode = 'single';        // 'single' | 'compare'
let datasetA = null;        // { rawData, dimensions, metrics, headerRow, period, fileName }
let datasetB = null;        // то же для файла сравнения
let compareMetric = null;   // активная метрика для дельты (по умолчанию — расход)
```

Существующие глобалы (`rawData`, `dimensions`, `metrics`, `headerRow`, `filteredData`, `charts`, `sortCol`, `sortDir`, `filterSelects`) сохраняются — это «активный датасет» для single-режима и для совместимости фильтров. В compare-режиме фильтры применяются к обоим датасетам (см. «Фильтры в compare-режиме»).

### `parseData` → чистая функция

Текущий `parseData(rows)` пишет напрямую в глобалы. Переписывается как чистая функция:

```js
function parseData(rows, fileName) {
  // возвращает { rawData, dimensions, metrics, headerRow, period, fileName }
  // не трогает глобалы
}
```

`handleFile` принимает параметр `target: 'A' | 'B'` и присваивает результат в `datasetA` или `datasetB`. Это убирает связанность парсинга с глобальным состоянием — улучшение, оправданное задачей.

`detectColumns(dataset)` и `aggregateBy(dimCol, dataset)` параметризуются датасетом вместо чтения глобалов.

### Поток сравнения

```
Тумблер включён
  → header показывает второй слот загрузки
  → handleFile(target) парсит файл в datasetA или datasetB
  → оба загружены → mode='compare' → renderCompare()
       ├→ aggregateBy(primaryDim, A) + aggregateBy(primaryDim, B)
       ├→ matchByLabel(a, b) — по primaryDim
       ├→ delta = B − A, deltaPct = (B−A)/A
       └→ renderDeltaKPIs / renderDeltaCharts / renderDeltaTable
```

## UX тумблера и навигация

### Размещение в header

```
[Яндекс.Директ — Дэшборд]   [○ Сравнение периодов]   [Файл A ▼]  [Загрузить Excel]
                                                              ↑ появляется при включении тумблера
```

Тумблер — стилизованный checkbox-switch в правой части header.

### Логика переключения mode

`mode` меняется автоматически по наличию данных, отдельной кнопки подтверждения нет:

- Тумблер **ВЫКЛ** (по умолчанию) → `single`. Слот «Файл A» скрыт, кнопка «Загрузить Excel» работает как сейчас. Ноль изменений в текущем UX.
- Тумблер **ВКЛ** + оба файла загружены → `compare`.
- Тумблер **ВКЛ** + только A → показываем A в обычном виде + баннер в `filtersBar`: «загрузите файл B для сравнения».
- Тумблер **ВКЛ** + только B → симметрично: баннер «загрузите файл A (основной)».
- Тумблер **ВЫКЛ** при загруженном B → `single`, datasetB остаётся в памяти, не используется. Повторное включение — мгновенно, без перезагрузки файлов.

### Состояния слотов

- Пустой → `Файл A: не загружен [Выбрать]`
- Загружен → `Файл A: master_june.xlsx ✓ [сменить]`
- Ошибка → toast + слот показывает `Файл B: ⚠ не распознан [сменить]`

### Подпись периода

В compare-режиме подпись периода в header: `1–31 мая 2026 → 1–30 июня 2026`. Период каждого файла извлекается текущей логикой парсинга (переиспользуется для обоих).

### Сохранение состояния

Не делаем (YAGNI). Тумблер стартует выключенным при загрузке страницы. В localStorage ничего не пишем.

## Рендер сравнения

Три блока переиспользуют существующих хелперов (`aggregateBy`, `chartOpts`, `barOpts`, `formatValue`, `PALETTE`, `makeChartCard`, `safeChart`) и добавляют слой дельты.

### renderDeltaKPIs()

KPI-сетка с дельтой. Те же приоритетные метрики, что в `renderKPIs` (Показы, Клики, Расход, CTR, Конверсии, Отказы, CPC/CPA…).

Каждая карточка:
- Значение **B** крупно.
- Строка дельты: `▲ +12.3% (1 200 → 1 348)`.
- Цвет строки по направлению метрики:
  - `goodWhenUp`: Показы, Клики, Конверсии, CTR — рост = зелёный, падение = красный.
  - `goodWhenDown`: Расход, CPC, CPA, Отказы — падение = зелёный, рост = красный.

Вводится словарь направлений в `STATE`/хелперах:

```js
const METRIC_DIRECTION = {
  impressions: 'up', clicks: 'up', conversions: 'up', ctr: 'up',
  spend: 'down', cpc: 'down', cpa: 'down', bounce: 'down'
};
```

Метрики сопоставляются со словарём по role (см. баг-фикс CTR).

### renderDeltaCharts()

Три целевых графика (не 8, чтобы не перегружать сравнение):

1. **Grouped bar** — значение A vs B по топ-15 кампаний по выбранной `compareMetric` (сортировка по сумме A+B, остальные кампании в «прочие»). Side-by-side, без двойной оси.
2. **Diverging bar** — дельта % по каждой кампании: влево (красный) — ухудшилось, вправо (зелёный) — улучшилось. Самый наглядный график сравнения.
3. **Scatter** — A по X, B по Y, диагональ y=x. Точки выше диагонали = рост, ниже = падение. Размер точки = расход.

### renderDeltaTable()

- Колонки: `Кампания | A | B | Δ абс. | Δ %`.
- Совпавшие кампании сверху, затем блок «Только в A» и «Только в B».
- Цветовое кодирование дельты по тем же правилам «хорошо/плохо».
- Сортировка кликом по любой колонке. Текущий `sortTable(col)` хранит один `sortCol`/`sortDir` и сортирует по числу или строке. В compare-таблице колонки — это `label`, `A`, `B`, `deltaAbs`, `deltaPct`; сортировка по числу работает как есть, по `label` — через `localeCompare`. Отдельной функции не нужно — `sortTable` остаётся, меняется только источник колонок.

## Матчинг кампаний

Точное совпадение `label` (значение primaryDim, обычно название кампании), нормализованное: `.toLowerCase()` + trim. Нечёткий поиск не делаем (YAGNI) — точное совпадение покрывает 99% случаев.

```js
function matchByLabel(rowsA, rowsB) {
  const norm = s => String(s).trim().toLowerCase();
  const mapB = new Map(rowsB.map(r => [norm(r.label), r]));
  const setA = new Set(rowsA.map(r => norm(r.label)));
  const matched = [], onlyA = [], onlyB = [];
  rowsA.forEach(a => {
    const b = mapB.get(norm(a.label));
    if (b) matched.push({ a, b }); else onlyA.push(a);
  });
  rowsB.forEach(b => { if (!setA.has(norm(b.label))) onlyB.push(b); });
  return { matched, onlyA, onlyB };
}
```

## Фильтры в compare-режиме

Фильтры (`buildFilters`) применяются к **обоим** датасетам по одним и тем же значениям primaryDim. Выбрана кампания «X» → показываем её и в A, и в B. Несовпадающие кампании в выборку не попадают. Отдельный UI фильтров для B не нужен.

В compare-режиме `buildFilters` строится по объединению значений измерений A и B.

## Баг-фиксы (минимум для корректности)

### 1. CTR-агрегация по role

Текущий код (`aggregateBy`, строки ~571–574) жёстко читает `'Показы_sum'` / `'Клики_sum'`:

```js
const imp = m['Показы_sum'] || 0;
const clk = m['Клики_sum'] || 0;
result[met.name] = imp ? (clk / imp * 100) : 0;
```

Ломается при иных названиях колонок. Решение: добавить role-метки в `COL_META`:

```js
'Показы': { type: 'metric', agg: 'sum', fmt: 'int', role: 'impressions' },
'Клики':  { type: 'metric', agg: 'sum', fmt: 'int', role: 'clicks' },
```

И искать по role: `metrics.find(m => m.meta.role === 'impressions')`. Применяется в `aggregateBy` и `renderKPIs` (строки ~620–623, там тот же хардкод `'Показы'`/`'Клики'`). Делает логику устойчивой к будущим переименованиям.

### 2. Toast-нотификации вместо alert()

Минимальная реализация:
- Фиксированный контейнер `#toasts` внизу справа, добавляется в DOM.
- Функция `toast(message, type)` (`success` / `error` / `info`) с авто-исчезновением через 5 секунд и кнопкой закрытия.
- Заменяет все 3 текущих `alert()` (битый файл, нет заголовков, нет метрик) и добавляется для ошибок файла B в compare-режиме.
- `alert` убирается полностью.

## Структура кода

Single-file `dashboard.html`. Блоки переупорядочиваются по зонам ответственности (разделители-комментарии уже есть, расширяем):

```
STATE           — mode, datasetA, datasetB, compareMetric + текущие глобалы
HELPERS         — hasValue, toNum, fmt, getMeta, formatValue + НОВЫЕ: roleBy(), toast()
FILE HANDLING   — handleFile(target), parseData(rows, fileName) → чистая функция
DETECT          — detectColumns(dataset) — параметризуется датасетом
AGGREGATION     — aggregateBy(dimCol, dataset) + НОВЫЕ: matchByLabel(), deltaRow()
RENDER SINGLE   — renderKPIs/Charts/Table (сейчас) — читают из активного датасета
RENDER COMPARE  — renderDeltaKPIs/Charts/Table (НОВЫЙ блок)
FILTRATION      — applyFilters — в compare работает по обоим датасетам
```

## Тестирование (вручную)

Бэкенда и тест-раннера нет, проверка ручная:

1. **Sanity:** файл A = июньский (`2026-06-20_...xlsx`), файл B = его копия → все дельты = 0, графики симметричны (scatter на диагонали).
2. **Ненулевая дельта:** файл B = модифицированная копия (вручную изменены пара значений расхода) → ненулевые дельты в ожидаемых кампаниях, diverging bar показывает направление верно.
3. **Смешанные форматы:** майский формат как A, июньский как B → матчинг по названию кампании работает (роль `campaign` определена для обоих названий).
4. **Битый B:** битый файл в слот B → toast с ошибкой, A остаётся активным.
5. **Переключение тумблера:** выкл/вкл → состояние восстанавливается без перезагрузки файлов.
6. **Single-режим без изменений:** загрузка одного файла при выключенном тумблере — прежнее поведение.
7. **«Только в A / только в B»:** файлы с разным набором кампаний → несоответствующие кампании попадают в соответствующие блоки таблицы.

## Критерии готовности

- Тумблер в header переключает single/compare без перезагрузки файлов.
- В compare-режиме показываются KPI с дельтой, 3 графика, таблица с Δ.
- Дельты считаются корректно (проверка сценарием 1 и 2).
- Матчинг кампаний точный, регистронезависимый.
- CTR считается по role, а не по хардкоду имени.
- `alert()` полностью заменён на toast.
- Single-режим визуально и функционально не изменён.
